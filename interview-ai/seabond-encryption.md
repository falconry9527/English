
# 信封加密 (Envelope Encryption)

# 一句话
```
用密钥加密密钥的两层方案：DEK 加数据，KEK 加 DEK。
大数据本地快加密，只把很小的 DEK 交给 KMS 托管。

对称加密  : 快,但密钥分发/管理难(双方得共享同一把密钥)
公钥用于加密或验证，私钥用于解密或签名

非对称加密 : 安全、好分发,但慢,大数据直接加会很卡

```

# 两层密钥
```
DEK (Data Encryption Key 数据密钥) : 直接加密真实数据，对称密钥(AES-256)，随机生成、用完即清。
KEK (Key Encryption Key 主密钥)   : 加密 DEK，存在 KMS/HSM 里，永不离开安全边界。
```

# 加密流程
```
1. 生成随机 DEK
2. DEK 加密明文数据        -> 密文
3. KEK 加密 DEK            -> 加密后的DEK(密信封)
4. 存储:  密文 + 加密后的DEK   (放在一起)
5. 内存里的明文 DEK 立即清除
```

# 解密流程
```
1. 取出 加密后的DEK
2. 请求 KMS 用 KEK 解密     -> 还原明文 DEK
3. 明文 DEK 解密密文        -> 原始数据
```

# 为什么叫"信封"
```
数据被 DEK 锁住，DEK 又被 KEK 锁住。
像把信(数据)放进信封(DEK)，信封再放进保险箱(KEK)，一层套一层。
```

# 优点
```
性能    : 大数据用本地对称 DEK 加密(快)，只把很小的 DEK 发给 KMS，网络/KMS 开销小。
密钥隔离: KEK 永不离开 KMS/HSM，泄露风险低。
轻松轮换: 轮换 KEK 只需重新加密很小的 DEK，无需重新加密海量数据。
细粒度  : 每个文件/每条记录用独立 DEK，单个 DEK 泄露影响范围有限。
分权管理: 加密数据的人不需要接触 KEK，权限分离。
```

# 实际应用
```
AWS KMS  : GenerateDataKey 一次返回 明文DEK + 加密后的DEK，典型信封加密。
GCP KMS / Azure Key Vault : 同理。
落地场景 : S3 服务端加密(SSE-KMS)、数据库字段加密、K8s Secrets 加密。
```

# 面试追问要点
```
Q: 为什么不直接用 KEK 加密数据?
A: KEK 在 KMS 里，每次加解密都要走网络调用 KMS，大数据量性能差、成本高；
   且 KEK 直接接触明文会扩大暴露面。DEK 在本地加密，快且 KEK 不出 KMS。

Q: 密钥轮换怎么做?
A: 轮换 KEK 时只需用新 KEK 重新加密那些小小的 DEK(re-wrap)，数据密文不动，秒级完成。

Q: DEK 泄露了怎么办?
A: 一个 DEK 只对应一份(或一批)数据，影响范围可控；KEK 没泄露，其他数据仍安全。
```
